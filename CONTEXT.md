# Intentive

Intentive is a macOS background service that captures what a user is doing on their computer, periodically compresses that activity into structured snapshots, and delivers them to an external agent. It is infrastructure, not a product — v1 has no behavioral intelligence of its own.

## Language

**Intentive**:
The macOS application and the project itself.
_Avoid_: app, client, wrapper

**ScreenPipe**:
The third-party open-source capture engine (Rust CLI binary) that Intentive bundles and manages. Responsible for recording screen, audio, and UI events and storing them in a local SQLite database.
_Avoid_: capture layer, recorder, engine (use ScreenPipe by name)

**Capture Session**:
The period during which ScreenPipe is actively recording — starts when the user opens Intentive and enables capture, ends when they stop or quit.
_Avoid_: recording session, active session

**Context Snapshot**:
A time-bounded, LLM-summarized record of what the user was doing during a 60-second activity window. Contains a prose summary generated by a local Ollama model and minimal metadata (id, captured_at, period_start, period_end). Intentive performs summarization on-device, writes the snapshot to a local SQLite log, then pushes to the OpenClaw Agent. Raw ScreenPipe data is consumed internally and not included in the payload or the log. Privacy guardrails are applied at summarization time via LLM prompt constraints.
_Avoid_: summary, context dump, activity log, raw snapshot

**Context Heartbeat**:
The internal service that runs on a 60-second cadence during a Capture Session. Fires only when meaningful activity has occurred since the last snapshot (new OCR text, audio transcription, or window switch detected via ScreenPipe's WebSocket). Queries ScreenPipe's local HTTP API and produces a Context Snapshot. Skips the cycle silently if no activity is detected.
_Avoid_: polling service, summarizer, heartbeat (use "Context Heartbeat" in full)

**OpenClaw Agent**:
The external agent that consumes Context Snapshots. Intentive treats it as a black box — the agent's internal behavior is out of scope for v1.
_Avoid_: agent, AI, LLM (use OpenClaw Agent when referring to the specific consumer)

**Agent Interface**:
The push mechanism by which Intentive delivers Context Snapshots to the OpenClaw Agent over HTTPS. Intentive POSTs each snapshot as a JSON payload to a configured webhook URL on the OpenClaw Agent's GCP VM. The agent wakes up on receipt.
_Avoid_: pull, polling, WebSocket, IPC

## Relationships

- **Intentive** bundles and manages one **ScreenPipe** process
- A **Capture Session** produces a stream of **Context Snapshots** via the **Context Heartbeat**
- Each **Context Snapshot** is derived from raw data stored in ScreenPipe's local SQLite
- **Context Snapshots** are delivered to the **OpenClaw Agent** via the **Agent Interface**

## Example dialogue

> **Dev:** "Should the Context Heartbeat query SQLite directly?"
> **Domain expert:** "No — it queries ScreenPipe's HTTP API. We only go to SQLite directly if the API doesn't expose what we need."

> **Dev:** "When does a Capture Session end?"
> **Domain expert:** "When the user stops capture from the menu bar or quits Intentive. ScreenPipe is shut down at that point."

**Auth**:
The mechanism that links a user's Intentive installation to their OpenClaw Agent endpoint. Signing in retrieves the webhook URL and API key for that user's agent. Capture runs without auth, but Context Snapshots are not delivered until the user is signed in.
_Avoid_: login, account (use "sign in" / "signed-in user")

## Flagged ambiguities

- "push vs pull" for the Agent Interface — resolved: **push**. Intentive delivers snapshots to the OpenClaw Agent. The agent is event-driven and wakes when a snapshot arrives. See ADR-0004.
- "agent" was used generically throughout early discussions — resolved: use **OpenClaw Agent** for the specific consumer, **Agent Interface** for the delivery mechanism.
- Auth provider (Supabase vs Neon vs other) — deliberately deferred. Decision depends on what database the OpenClaw Agent backend uses; both projects are being built in parallel and must be compatible. Auth is the last thing wired up in v1.
